Descripción de Pertinencia de la Arquitectura de la Base de Datos

Tabla Clientes

Propósito

La tabla Clientes es fundamental para almacenar la información de los usuarios que interactúan con el sistema. Permite gestionar la identidad y los datos de contacto de cada cliente, asegurando una comunicación efectiva y un acceso seguro a sus cuentas.

Campos

  • ClienteID: Clave primaria única que identifica a cada cliente.
  • PrimerNombre, SegundoNombre, ApellidoPaterno y ApellidoMaterno: Permiten un tratamiento completo y formal del cliente.
  • CorreoElectronico: Único y obligatorio, permite la autenticación y comunicación efectiva.
  • Contraseña: Almacena de forma segura la contraseña del cliente, necesaria para el acceso a su cuenta.
  • Telefono: Campo obligatorio que facilita el contacto directo con el cliente.
  • Calle, NumeroExterior, NumeroInterior, Colonia, Municipio, CodigoPostal: Campos que permiten registrar la dirección de entrega, esencial para el procesamiento de pedidos.
  • UltimoAcceso: Permite realizar análisis sobre la actividad del cliente, útil para la gestión de cuentas inactivas.

Tabla Pedidos

Propósito

La tabla Pedidos registra todos los pedidos realizados por los clientes, permitiendo gestionar el historial de compras y el estado actual de cada transacción.

Campos

  • PedidoID: Clave primaria única que identifica cada pedido.
  • ClienteID: Clave foránea que relaciona el pedido con el cliente que lo realizó.
  • FechaPedido: Registra la fecha en que se realizó el pedido, esencial para el seguimiento.
  • EstadoPedido: Indica el estado actual del pedido (Pendiente, En Proceso, Entregado, Cancelado), facilitando la gestión operativa.
  • Total: Monto total del pedido, necesario para la contabilidad y el análisis de ventas.
  • FechaEntregaEstimada: Proporciona una fecha estimada para la entrega, ayudando a gestionar las expectativas del cliente.

Tabla Comentarios

Propósito

La tabla Comentarios permite a los clientes dejar comentarios, preguntas o reseñas sobre los productos, fomentando la interacción y retroalimentación.

Campos

  • ComentarioID: Clave primaria única que identifica cada comentario.
  • ClienteID: Clave foránea que vincula el comentario al cliente que lo realizó.
  • Comentario: Contenido del comentario, que puede ser una pregunta, opinión o reseña.
  • FechaComentario: Registra la fecha en que se realizó el comentario, útil para el seguimiento de interacciones.

Tabla Respuestas

Propósito

La tabla Respuestas almacena las respuestas proporcionadas por los propietarios o administradores a los comentarios de los clientes, facilitando la atención al cliente.

Campos

  • RespuestaID: Clave primaria única que identifica cada respuesta.
  • Respuesta: Contenido de la respuesta, que aborda la consulta o comentario del cliente.
  • FechaRespuesta: Registra la fecha en que se realizó la respuesta, ayudando a mantener un historial de interacciones.

Tabla ComentarioRespuesta

Propósito

La tabla ComentarioRespuesta actúa como una tabla de unión para gestionar las relaciones entre comentarios y respuestas, permitiendo múltiples respuestas a un solo comentario.

Campos

  • ComentarioID: Clave foránea que vincula la relación al comentario correspondiente.
  • RespuestaID: Clave foránea que vincula la relación a la respuesta correspondiente.
  • Clave primaria (ComentarioID, RespuestaID): Asegura que cada combinación de comentario y respuesta sea única, evitando duplicados en las relaciones.

La arquitectura de la base de datos, a través de sus tablas y campos, está diseñada para satisfacer las necesidades del sistema mientras asegura la integridad y la eficiencia de datos. Cada tabla cumple un propósito específico y sus campos están alineados con los requerimientos funcionales de los clientes y propietarios.

Normalización de la Base de Datos

La base de datos ha sido diseñada siguiendo las primeras tres formas normales (1NF, 2NF, 3NF) para garantizar la organización y consistencia de los datos:

1NF (Primera Forma Normal)

Cada tabla tiene una clave primaria única y todas las entradas son atómicas, lo que significa que cada campo contiene un solo valor. Por ejemplo:

  • Tabla Clientes: Contiene campos como PrimerNombre, SegundoNombre, ApellidoPaterno y ApellidoMaterno, donde cada uno de estos campos guarda un solo valor. ClienteID actúa como clave primaria, identificando de manera única a cada cliente.
  • Tabla Pedidos: Cada pedido es único gracias al campo PedidoID, y todos los campos como FechaPedido y EstadoPedido contienen valores individuales sin listas ni conjuntos.
  • Tabla Comentarios: Cada comentario tiene un ComentarioID único, y los campos como Texto y FechaComentario son atómicos.
  • Tabla Respuestas: Con RespuestaID como clave primaria, cada respuesta se almacena de forma independiente y todos los campos son atómicos.
  • Tabla ComentarioRespuesta: Esta tabla de unión contiene únicamente las claves foráneas ComentarioID y RespuestaID, asegurando que cada relación sea única y sin duplicados.

2NF (Segunda Forma Normal)

La base de datos está en 1NF y todos los atributos no clave dependen completamente de la clave primaria. Por ejemplo:

  • Tabla Clientes: Todos los campos, como PrimerNombre, ApellidoPaterno y CorreoElectronico, dependen completamente de ClienteID.
  • Tabla Pedidos: Cada atributo, incluyendo ClienteID, FechaPedido, EstadoPedido y Total, depende completamente de PedidoID.
  • Tabla Comentarios: Todos los campos dependen únicamente de ComentarioID, garantizando que no haya dependencia parcial.
  • Tabla Respuestas: Todos los campos son dependientes de RespuestaID, sin dependencias parciales.
  • Tabla ComentarioRespuesta: La combinación de ComentarioID y RespuestaID como clave primaria asegura que todos los campos dependan completamente de esta clave compuesta.

3NF (Tercera Forma Normal)

La base de datos está en 2NF y no hay dependencias transitivas entre los atributos no clave. Por ejemplo:

  • Tabla Clientes: Todos los atributos no clave dependen únicamente de ClienteID, sin que ningún atributo no clave dependa de otro.
  • Tabla Pedidos: Todos los campos dependen solo de PedidoID, y no existen atributos no clave que dependan de otros atributos no clave.
  • Tabla Comentarios: Los campos dependen únicamente de ComentarioID, sin dependencias transitivas.
  • Tabla Respuestas: Todos los campos dependen únicamente de RespuestaID, garantizando que no haya dependencias transitivas.
  • Tabla ComentarioRespuesta: No hay atributos no clave, ya que solo contiene las claves foráneas, lo que elimina cualquier posibilidad de dependencia transitiva.

Este diseño normalizado asegura una estructura de datos eficiente, minimizando redundancias y facilitando la gestión de la información en la base de datos.

Código SQL para crear tablas


        CREATE DATABASE FlosrisoftBD
        USE FlosrisoftBD

        CREATE TABLE Clientes (
            ClienteID INT PRIMARY KEY IDENTITY,
            PrimerNombre NVARCHAR(30) NOT NULL,
            SegundoNombre NVARCHAR(30) NULL,
            ApellidoPaterno NVARCHAR(30) NOT NULL,
            ApellidoMaterno NVARCHAR(30) NOT NULL,
            CorreoElectronico NVARCHAR(50) UNIQUE NOT NULL,
            Contraseña NVARCHAR(30) NOT NULL,
            Telefono NVARCHAR(15) NOT NULL,
            Calle NVARCHAR(50) NOT NULL,
            NumeroExterior NVARCHAR(10) NOT NULL,
            NumeroInterior NVARCHAR(10) NULL,
            Colonia NVARCHAR(50) NOT NULL,
            Municipio NVARCHAR(50) NOT NULL,
            Referencias NVARCHAR(100) NULL,
            CodigoPostal NVARCHAR(10) NOT NULL,
            UltimoAcceso DATETIME NOT NULL
        )

        CREATE TABLE Pedidos (
            PedidoID INT PRIMARY KEY IDENTITY,
            ClienteID INT NOT NULL FOREIGN KEY REFERENCES Clientes(ClienteID), 
            FechaPedido DATETIME NOT NULL DEFAULT GETDATE(),
            EstadoPedido NVARCHAR(50) NOT NULL,  -- Ejemplo: Pendiente, En Proceso, Entregado, Cancelado
            Total DECIMAL(10, 2) NOT NULL,
            FechaEntregaEstimada DATETIME NULL
        )

        CREATE TABLE Comentarios (
            ComentarioID INT PRIMARY KEY IDENTITY,
            ClienteID INT NOT NULL FOREIGN KEY REFERENCES Clientes(ClienteID),
            Comentario NVARCHAR(255) NOT NULL,
            FechaComentario DATETIME NOT NULL DEFAULT GETDATE()
        )

        CREATE TABLE Respuestas (
            RespuestaID INT PRIMARY KEY IDENTITY,
            Respuesta NVARCHAR(255) NOT NULL,
            FechaRespuesta DATETIME NOT NULL DEFAULT GETDATE()
        )

        CREATE TABLE ComentarioRespuesta (
            ComentarioID INT NOT NULL FOREIGN KEY (ComentarioID) REFERENCES Comentarios(ComentarioID),
            RespuestaID INT NOT NULL FOREIGN KEY (RespuestaID) REFERENCES Respuestas(RespuestaID),
            PRIMARY KEY (ComentarioID, RespuestaID)  -- Combinación de claves foráneas como clave primaria
        )
